# 1주차 (6.29~7.5) - Ch1,2: 개요, 실습 환경 구성(GKE)

### Chapter 1. AI 시대, 개발자의 인프라

Chapter 1은 "왜 지금 개발자가 인프라를, 그것도 AI와 함께 다뤄야 하는가"를 설명하는 도입부다. 책 전체의 큰 그림과 실습 시나리오를 소개한다.

#### 1.1 개발자에게 인프라가 다가온 시대
과거엔 인프라 팀이 서버를 만들고 개발자는 "배포해주세요"라고 요청만 했다. 지금은 그 경계가 무너졌고, 배경은 세 가지다.
- **1.1.1 DevOps 문화의 확산** — "You build it, you run it". 개발자가 코드를 짜는 것에서 끝나지 않고 배포·모니터링·장애 대응까지 책임진다.
- **1.1.2 클라우드 네이티브 전환** — 서버에 SSH로 접속해 수동 설정하는 대신, YAML이나 Terraform 같은 코드로 인프라를 선언하고 Git으로 버전 관리한다.
- **1.1.3 풀스택의 범위 확장** — 개발자가 알아야 할 범위가 프론트엔드/백엔드를 넘어 배포 파이프라인, 쿠버네티스 매니페스트, Helm 차트, CI/CD, 모니터링 설정까지 넓어졌다.

#### 1.2 쿠버네티스, 클라우드 인프라의 공통 언어
- **1.2.1 왜 쿠버네티스인가** — 특정 클라우드에 종속되지 않는다(AWS EKS, GCP GKE, Azure AKS, 온프레미스 모두 동일하게 동작). 클라우드 세계의 "운영체제" 같은 존재.
- **1.2.2 가파른 학습 곡선** — 도구 선택지가 너무 많은 게 진입 장벽이다.

  | 영역 | 선택지 |
  |---|---|
  | 모니터링 | Prometheus vs Datadog vs New Relic vs Grafana Cloud |
  | 로그 수집 | ELK(Elasticsearch+Logstash+Kibana) vs Loki+Fluent Bit |
  | 배포 전략 | Rolling Update vs Blue/Green vs Canary |
  | GitOps 도구 | ArgoCD vs Flux |
  | 시크릿 관리 | Sealed Secrets vs External Secrets vs CSI Driver |
  | 메시지 큐 | Kafka vs RabbitMQ vs NATS |
  | 트래픽 관리 | Ingress vs Gateway API |
- **1.2.3 AI라는 새로운 동료** — 이 수많은 선택과 설정을 AI와 함께 풀어나간다.

#### 1.3 GitOps에서 GitAIOps로
- **1.3.1 GitOps: 선언적 관리의 시작** — Git을 단일 진실 공급원 으로 삼아, 원하는 상태를 Git에 선언하면 ArgoCD/Flux 같은 도구가 클러스터를 그 상태로 자동으로 맞춘다.
- **1.3.2 GitOps만으로는 부족한 이유** — 장점은 크지만, 정작 YAML을 사람이 직접 작성해야 한다. 리소스 종류가 많고 설정이 복잡하다.
- **1.3.3 AI가 채우는 빈자리** — AI가 사람이 하던 손 작업을 대신한다: ①**매니페스트 작성**(자연어→YAML) ②**트러블슈팅**(로그·이벤트 분석→원인 제시) ③**문서화**(작업 과정 자동 기록).
- **1.3.4 GitAIOps: Git + AI + Ops** — Git + AI + Ops의 결합.

  | 구분 | GitOps | GitAIOps |
  |---|---|---|
  | 상태 정의 | 사람이 YAML 작성 | AI가 자연어로부터 YAML 생성 |
  | 배포 | 깃 푸시 → 자동 Sync | 동일 |
  | 트러블슈팅 | 사람이 로그 분석 | AI가 로그 분석 후 수정안 제시 |
  | 문서화 | 사람이 별도 작성(또는 안 함) | AI가 작업과 동시에 문서 생성 |
  | 검증 | 사람이 수동 확인 | AI가 클러스터 상태와 문서를 비교 |

#### 1.4 이 책의 구성과 실습 흐름
- **1.4.1 이 책의 흐름** — 이론서가 아니라 실습 위주. 하나의 서비스가 성장하는 과정을 9개 장에 걸쳐 따라간다.

  | 구간 | 장 | 핵심 |
  |---|---|---|
  | 도입 | 1장 | AI 시대, 개발자의 인프라 |
  | 환경 구성 | 2장 | GKE + 클라우드 코드 + 첫 배포 |
  | SMB | 3~5장 | GitOps, CI, 관측 가능성, 무중단 배포 |
  | 전환기 | 6장 | 캐시, 보안, Canary |
  | 엔터프라이즈 | 7~8장 | 규모 확장, 고도화 |
  | 종합 | 9장 | GitAIOps: 살아있는 운영 표준 |
- **1.4.2 실습 환경** — 클라우드: GKE, AI: 클로드 코드, 형상 관리: 깃허브.
- **1.4.3 이 책의 저장소** — 두 개: `_Book_GitAIOps`(가이드·의사결정·정답 예시), `notiflex-platform`(독자가 직접 실습하며 채워가는 저장소).

#### 1.5 Notiflex 스타트업 시나리오 소개
- **1.5.1 Notiflex란?** — 실습을 관통하는 가상의 B2B 알림 SaaS 플랫폼. 이메일/SMS/푸시 알림을 대신 발송해주는 서비스. Go로 작성된 API 서버 중심.
- **1.5.2 성장 타임라인** — 창업(2장) → 첫 고객(3장) → 서비스 장애(4장) → 성장 시작(5장) → 전환기(6장) → 대형 계약(7장) → 대규모 운영(8장) → 돌아보기(9장). 각 성장 단계의 문제를 인프라로 해결한다.
- **1.5.3 여러분의 역할** — 독자는 Notiflex의 DevOps 엔지니어. 클로드 코드를 도구로 삼아 서비스 성장에 맞춰 인프라를 키운다.

#### 1.6 가드레일: 클로드 코드가 정확하게 동작하는 이유
- **1.6.1 CLAUDE.md와 가드레일** — 클로드 코드는 저장소 루트의 `CLAUDE.md`를 자동으로 읽어 동작한다. 가드레일 3종: ①**의사결정 가이드**(decision-guides) ②**실행 가드레일**(prompt-guardrails) ③**결과 템플릿**(result-templates).
- **1.6.2 3단계 흐름: 탐색 → 비교 → 실행**
- **1.6.3 왜 가드레일이 필요한가** — AI가 강력해도 정확성이 중요하다. 가드레일은 AI가 정해진 범위 안에서, 검증된 경로를 따라 동작하게 만드는 시스템. 잘못된 값 하나가 클러스터 전체에 영향을 줄 수 있기 때문.

### Chapter 2. 환경 구성

Chapter 2는 이후 9개 장에서 쓸 토대를 실제로 세팅하는 실습 장이다. 

#### 2.2 클로드 코드 설치
- 클로드 코드 설치와 첫 실행은 직접 했다(`irm https://claude.ai/install.ps1 | iex`). `_Book_GitAIOps`를 clone하고 `claude --dangerously-skip-permissions`로 실행했는데, 이 모드는 승인을 안 물어봐서 이후 명령을 클로드 코드가 자동 실행한다. 
- statusline 설정을 지시했다. 클로드 코드가 스크립트를 구성하다가 윈도우에서 두 번 막혔고 클로드 코드를 통해 해결했다. winget으로 설치한 `jq`가 Git Bash에서 실행되지 않아 실제 `jq.exe` 경로를 찾는 리졸버를 넣었다고 한다.

#### 2.3 gcloud CLI 설치
- gcloud 설치·인증·기본값 설정을 지시했다. 클로드 코드가 설치·설정했고, 브라우저 인증만 직접 했다. kubectl은 gcloud에 묶여 설치되는데, 기존에 사용하던 버전이 PATH 설정에서 우선되면서 문제가 됐다.
- 우선순위 조정 중 PATH가 통째로 재작성돼 사고가 났다. 클로드 코드가 관리자 권한으로 복구했고, 'PATH 전체 재작성 금지'를 규칙으로 삼도록 지시했다.

#### 2.4 깃허브 저장소 구성
- 저장소 구성을 지시했다. 클로드 코드가 `.gitignore`와 디렉터리 구조, 그리고 `CLAUDE.md`를 만들었다.

#### 2.5 GKE 클러스터 생성
- 클러스터 생성을 지시했다. 클로드 코드가 아래 명령으로 생성했다. 실무에선 까다롭게 느꼈던 부분인데 아무런 이슈가 없었다.

#### 2.6 Notiflex 앱 빌드와 배포
- 앱과 컨테이너 작성을 지시했다.
- 빌드·푸시를 지시했다. 이때 `PERMISSION_DENIED`가 났는데, 신규 프로젝트라 Compute 기본 서비스 계정에 빌더 권한이 없었다. 클로드 코드가 원인을 진단해 `roles/cloudbuild.builds.builder`를 부여해 해결했다.
- 클로드 코드가 매니페스트 3종을 작성했다.
- 배포·검증을 지시했다. 클로드 코드가 몇번 단답형으로만 검증결과를 알려줬다. 좀더 Raw한 형태의 답변을 받기위해 몇차례 재질문을 했고 kubectl 실행시에는 리턴되는 값을 그대로 보여주도록 했다.
- 앱에 관련해서도 권한이슈 이외에는 딱히 이슈가 없어 구성을 자세히 살펴보지 않았다.

#### 2.7 깃허브에 첫 커밋
- 커밋·push를 지시했다. 클로드 코드가 코드·매니페스트를 `main`에 커밋하고 `JOURNEY.md`를 만들었다.
- push에서 두 번 막혔다. ①로컬 기본 브랜치가 `master`인데 원격은 `main`이라 refspec이 안 맞아 클로드 코드가 `git branch -m master main`으로 통일했다. ②토큰에 `workflow` 스코프가 없어 `.github/workflows/` push가 거부돼 자리표시자를 빼고 push했다.

#### 2.8 마무리: /update-docs 스킬
- 작업 내용을 문서에 자동 반영하는 커스텀 스킬 생성했다.

#### 2.9 2장 가드레일 살펴보기
- 2장은 각 절마다 `prompt-guardrails`(실행 지침)만 사용한다.
- 환경 구성 단계라 도구를 선택할 일이 없어서 "탐색→비교" 없이 실행 지침만 있다.

겪은 이슈와 해결

| 단계  | 문제                                                    | 해결                                                          |
| --- | ----------------------------------------------------- | ----------------------------------------------------------- |
| 2.2 | winget으로 깐 `jq`가 Git Bash에서 실행 안 됨(앱 실행 별칭)           | statusline 스크립트에 실제 `jq.exe` 경로 리졸버 추가                      |

| 2.6 | Cloud Build `PERMISSION_DENIED`(신규 프로젝트, 기본 SA 권한 부족) | Compute 기본 SA에 `roles/cloudbuild.builds.builder` 부여         |

## 느낀점 & 회고

**1. 중간중간 이슈를 어떻게 파악하고 결정하나?**
에러를 읽고 → 가드레일의 `트러블슈팅` 섹션과 대조 → 행동 전 현재 상태 확인(`kubectl get`, `git status`). CLAUDE.md 행동 규칙에 "변경 전 상태 확인"이 명시돼 있다.

**2. 내 업무도 가드레일처럼 구성해 신규 입사자에게 전달하면 좋겠다**
`CLAUDE.md`(규칙) + `prompt-guardrails`(실행 절차) + `decision-guides`(판단 기준) + `result-templates`(검증)가 곧 온보딩 시스템이다. Claude Code가 `CLAUDE.md` + `.claude/commands/`로 이를 지원하고, 9장에 온보딩 문서 자동 생성도 있다.

**3. 실습은 내가 하는 게 별로 없었다**
클로드 코드가 생성하는 인프라, 애플리케이션에 대해서 분석하는 시간이 더 필요하다.

**4. [필수입력]이 어떤 건 짧고 어떤 건 긴 이유가 무엇일까?**
`[필수입력]`은 사람이 치는 입력이고, 가드레일 파일은 AI가 참조하는 실행서라 역할이 다르다.
- 짧은 것은 실행 트리거("커밋하고 푸시해줘"). 상세는 가드레일 파일이 갖고 있다.
- 긴 것은 탐색 질문. 독자의 상황·맥락을 담아서 길어진다.

**5. [궁금하면]은 어떻게 구성돼 있을까?**
비교 단계다. `decision-guides`의 `## 비교` 섹션이 근거이며 건너뛰어도 된다. 각 가드레일 끝 `질문해보기` 블록이 궁금하면 던질 질문 예시다.
