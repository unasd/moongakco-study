### Chapter 3. 첫 번째 배포 파이프라인

푸시 방식 배포의 한계에서 출발해, GitOps로 "깃에 선언하면 클러스터가 알아서 맞추는" 풀 방식으로 넘어가고, CI와 연결해 코드 push만으로 배포까지 자동화한다. 실행은 클로드 코드가 하고, 나는 지시·판단·개입을 맡았다.

#### 3.1 푸시 기반 배포의 한계
- 명령형은 클러스터에 직접 손대는 방식이라, 실제 상태가 Git·문서와 어긋나는 configuration drift가 생긴다. 누가 언제 왜 바꿨는지 추적도 안 된다.
- 선언형은 원하는 상태를 Git에 적어두고 클러스터가 그 상태로 수렴하게 한다. GitOps는 이 선언형 + Git을 단일 기준으로 삼는 방식이다.

#### 3.2 ArgoCD 설치·GitOps 연결
- ArgoCD 설치와 GitHub 저장소 연결을 지시. 클로드 코드가 v3.4.4를 설치하고 SSH Deploy Key로 repo를 연결했다. admin 초기 비번은 시크릿에서 확인 후 변경.
- ArgoCD v3 stable manifest의 NetworkPolicy가 repo-server의 GitHub 외부 접근을 막을 수 있다는 이슈가 있었는데, 선제로 지우지 않고 Sync를 먼저 시도해 정상 통과를 확인했다.

#### 3.3 롤링 업데이트: 깃 푸시만으로 배포
- `/version` 응답 기능을 추가하고 v0.1.1로 배포하도록 지시. 매니페스트를 고쳐 Git에 push하니 ArgoCD가 감지해 롤링 업데이트했다. `git revert`로 되돌리면 이전 버전으로 롤백되는 것까지 시연했다.

#### 3.4 GitHub Actions CI: 빌드 자동화
- push 이벤트로 이미지를 빌드·푸시하는 CI를 지시. 클로드 코드가 워크플로를 만들고 이미지에 git SHA 태그를 붙였다.
- GCP 인증용 SA JSON 키 발급이 조직 정책(`iam.disableServiceAccountKeyCreation`)에 막혔다. org 레벨 권한으로 정책을 해제한 뒤 발급했다. 키 없이 가는 Workload Identity Federation이 더 정석이다.
- `.github/workflows/` push에 workflow 스코프가 필요해 `gh auth refresh -s workflow`로 인증했다.

#### 3.5 CI + ArgoCD 연결: 빌드→배포 자동화
- CI가 빌드 후 매니페스트의 이미지 태그를 갱신·push하고, 그걸 ArgoCD가 감지해 자동 배포하도록 연결. 코드만 고쳐 push하면 배포까지 이어지는 것을 e2e로 검증했다.
- CI 봇이 매니페스트를 push하려면 repo 쓰기 권한이 필요해 `contents: write`를 부여했다. CI가 SHA 태그 커밋을 만들어 로컬이 뒤처지므로 배포 후 `git pull`로 동기화했다.

#### 3.6 마무리: CLAUDE.md 행동 규칙 추가
- 상태 확인, 명령 출력 그대로 보여주기 등의 행동 규칙을 `CLAUDE.md`에 추가했다.

#### 도구 선택
| 영역 | 선택 | 검토한 대안 | 이유 |
|---|---|---|---|
| GitOps CD | ArgoCD | Flux, Jenkins X, Spinnaker | Web UI 배포 시각화, Application CRD, Self-Heal, 2노드에서 구동 가능 |

### Chapter 4. 관측 가능성 한 번에 구축하기

관측 가능성은 시스템 상태를 메트릭·로그·트레이스로 파악하는 것이다. 4장은 이 중 메트릭·로그·알림을 한 번에 구축한다. 트레이스는 8장이다. 실행은 클로드 코드가, 지시·판단·개입은 내가 맡았다.

#### 4.1 관측 가능성이란
- 메트릭: 숫자다. CPU 90%, 초당 요청 500, 에러율 2% 같은 값으로 지금 상태가 정상인지 판단한다.
- 로그: 텍스트다. 에러 메시지로 무엇이 잘못됐는지 원인을 찾는다.
- 트레이스: 흐름이다. 요청이 어느 서비스에서 느려졌는지 추적한다. 8장에서 추가한다.
- 이게 없으면 새벽에 장애가 나도 `kubectl get pods`·`logs`로 일일이 확인해야 하고, Pod이 재시작되면 이전 로그도 사라진다.

#### 4.2 메트릭 모니터링: Prometheus + Grafana
- Helm으로 kube-prometheus-stack을 설치해 Prometheus·Grafana·Alertmanager를 한 번에 올리도록 지시. 클로드 코드가 설치하고 Grafana 접속·대시보드까지 확인해줬다.
- 버전: Prometheus v3.13.1, Grafana 13.1.0, Alertmanager v0.33.1

#### 4.3 로그 수집: Loki + Fluent Bit
- Loki를 SingleBinary로, Fluent Bit를 DaemonSet으로 설치하도록 지시. 여기서 연결이 여러 번 막혔고, 대부분 클로드 코드가 원인을 잡아 해결했다.
  - Fluent Bit 기본 Loki 주소 `fluent-bit-loki`와 경로 `/api/prom/push`가 틀려서 `loki.serviceName=loki`, `servicePath=/loki/api/v1/push`로 교정. 구 경로는 이미 제거된 것이었다.
  - 내 config 블록이 차트 스키마와 안 맞아 무시됨 → `helm show values`로 스키마 확인 후 올바른 키만 override
  - Loki 데이터소스가 Grafana에 안 떠서 `grafana_datasource=1` 라벨을 붙인 ConfigMap을 별도 생성
  - Fluent Bit tail이 startup 이전 로그를 안 읽어 notiflex 로그가 안 보였다. Pod을 재생성해 새 startup 로그를 유발

#### 4.4 알림 설정: PrometheusRule
- Pod 재시작이 잦으면 알리는 PrometheusRule + Alertmanager를 지시. 클로드 코드가 규칙을 만들었다. 규칙이 CRD라 Git으로 관리돼 GitOps와 호환된다.
- 검증이 까다로웠다. notiflex가 scratch 이미지라 `exec`로 프로세스를 kill할 수 없고, `delete`는 restarts=0이라 알림 조건인 재시작 횟수를 못 맞춘다. 크래시루프를 내는 임시 Pod을 띄워 restarts>2를 유발해 알림이 firing되는 것을 확인했다.

#### 4.5 마무리: 메모리에 작업 컨텍스트 기록
- 이번 장 작업 컨텍스트를 기록.

#### 도구 선택
| 영역 | 선택 | 검토한 대안 | 이유 |
|---|---|---|---|
| 메트릭 | Prometheus + Grafana | Datadog, CloudWatch, Cloud Monitoring | 오픈소스 표준, 무료, e2-medium 구동 가능, Grafana로 로그·트레이스 통합 |

## 느낀점 & 회고

**1. sed 태그 교체 vs kustomize**
CI가 이미지 태그를 `sed`로 바꾸는 방식을 쓰는데, 이건 실무에서 쓰는 방식과 같다. sed는 문자열 치환이라 깨지기 쉬워서, ArgoCD가 네이티브로 지원하는 kustomize(`images:` 트랜스포머)로 태그를 관리하는 방식이 더 깔끔하다. 그 도입이 기대된다.

**2. 프로그램적 통제장치의 필요**
2장에선 실습이 수동적이라 느꼈는데, 3장에선 반대로 AI가 위험 명령 `kubectl delete`를 결국 실행하는 걸 봤다. 프롬프트 수준 가드레일만으론 부족하고, 프로그램적 통제장치가 필요하다고 느꼈다. 책은 이 필요를 7장 `settings.local.json`으로 다룬다. 위험 명령은 차단하고 비용 드는 명령은 승인을 받게 하는 방식이다.

**3. Prometheus·Loki 각 모듈의 역할을 이해하게 됐다**
Prometheus는 메트릭 수집, Grafana는 시각화, Loki는 로그 저장, Fluent Bit는 로그 수집, Alertmanager는 알림을 맡는다. 각각이 무슨 일을 하는지 잘 몰랐는데, 직접 구축하며 역할이 정리됐다.

**4. 회사 EFK와 비교 — 리소스 선정 기준을 다시 보게 됐다**
회사에선 EFK를 쓰고, Prometheus·Loki 도입을 여러 번 시도했지만 매번 리소스 부족으로 막혔다. 이번에 SingleBinary 모드 Loki와 Fluent Bit가 e2-medium 2노드 같은 작은 자원에서도 도는 걸 보고, 과거 시도의 리소스 선정 기준을 다시 검토해볼 수 있게 됐다.
