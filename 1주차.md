# 1주차 (6.29~7.5) - Ch1,2: 개요, 실습 환경 구성(GKE)

### Chapter 1. AI 시대, 개발자의 인프라

Chapter 1은 "왜 지금 개발자가 인프라를, 그것도 AI와 함께 다뤄야 하는가"를 설명하는 도입부다. 책 전체의 큰 그림(GitAIOps)과 실습 시나리오(Notiflex)를 소개한다.

#### 1.1 개발자에게 인프라가 다가온 시대
과거엔 인프라 팀이 서버를 만들고 개발자는 "배포해주세요"라고 요청만 했다. 지금은 그 경계가 무너졌고, 배경은 세 가지다.
- **1.1.1 DevOps 문화의 확산** — "You build it, you run it"(만든 사람이 운영한다). 개발자가 코드를 짜는 것에서 끝나지 않고 배포·모니터링·장애 대응까지 책임진다.
- **1.1.2 클라우드 네이티브 전환** — 서버에 SSH로 접속해 수동 설정하는 대신, YAML이나 Terraform 같은 **코드로 인프라를 선언**하고 Git으로 버전 관리한다(IaC).
- **1.1.3 풀스택의 범위 확장** — 개발자가 알아야 할 범위가 프론트엔드/백엔드를 넘어 배포 파이프라인, 쿠버네티스 매니페스트, Helm 차트, CI/CD, 모니터링 설정까지 넓어졌다.

#### 1.2 쿠버네티스, 클라우드 인프라의 공통 언어
- **1.2.1 왜 쿠버네티스인가** — 특정 클라우드에 종속되지 않는다(AWS EKS, GCP GKE, Azure AKS, 온프레미스 모두 동일하게 동작). 클라우드 세계의 "운영체제" 같은 존재. 핵심 개념: `Deployment/Pod`(배포·실행), `Service/Ingress`(트래픽 라우팅), `ConfigMap/Secret`(설정·민감정보), `HPA`(자동 확장), 모니터링·로깅.
- **1.2.2 가파른 학습 곡선** — 도구 선택지가 너무 많은 게 진입 장벽이다.

  | 영역 | 선택지 |
  |---|---|
  | 모니터링 | Prometheus vs Datadog vs New Relic vs Grafana Cloud |
  | 로그 수집 | ELK(Elasticsearch+Logstash+Kibana) vs Loki+Fluent Bit |
  | 배포 전략 | Rolling Update vs Blue/Green vs Canary |
  | GitOps 도구 | ArgoCD vs Flux |
  | 시크릿 관리 | Sealed Secrets vs External Secrets vs CSI Driver |
  | 메시지 큐 | Kafka vs RabbitMQ vs NATS |
  | 트래픽 관리 | Ingress vs Gateway API |
- **1.2.3 AI라는 새로운 동료** — 이 수많은 선택과 설정을 AI(클로드 코드)와 함께 풀어나간다.

#### 1.3 GitOps에서 GitAIOps로
- **1.3.1 GitOps: 선언적 관리의 시작** — Git을 **단일 진실 공급원(Single Source of Truth)** 으로 삼아, 원하는 상태를 Git에 선언하면 ArgoCD/Flux 같은 도구가 클러스터를 그 상태로 자동으로 맞춘다(Push가 아니라 감지·동기화).
- **1.3.2 GitOps만으로는 부족한 이유** — 장점(배포 자동화, 배포 히스토리, 롤백=git revert, 코드 리뷰=PR, 재현 가능성)은 크지만, 정작 **YAML을 사람이 직접 작성**해야 한다. 리소스 종류가 많고 설정이 복잡하다.
- **1.3.3 AI가 채우는 빈자리** — AI가 사람이 하던 손 작업을 대신한다: ①**매니페스트 작성**(자연어→YAML) ②**트러블슈팅**(로그·이벤트 분석→원인 제시) ③**문서화**(작업 과정 자동 기록).
- **1.3.4 GitAIOps: Git + AI + Ops** — Git(선언·기록) + AI(생성·분석) + Ops(운영)의 결합.

  | 구분 | GitOps | GitAIOps |
  |---|---|---|
  | 상태 정의 | 사람이 YAML 작성 | AI가 자연어로부터 YAML 생성 |
  | 배포 | 깃 푸시 → 자동 Sync | 동일 |
  | 트러블슈팅 | 사람이 로그 분석 | AI가 로그 분석 후 수정안 제시 |
  | 문서화 | 사람이 별도 작성(또는 안 함) | AI가 작업과 동시에 문서 생성 |
  | 검증 | 사람이 수동 확인 | AI가 클러스터 상태와 문서를 비교 |

#### 1.4 이 책의 구성과 실습 흐름
- **1.4.1 이 책의 흐름** — 이론서가 아니라 **실습 위주**. 하나의 서비스(Notiflex)가 성장하는 과정을 9개 장에 걸쳐 따라간다.

  | 구간 | 장 | 핵심 |
  |---|---|---|
  | 도입 | 1장 | AI 시대, 개발자의 인프라 |
  | 환경 구성 | 2장 | GKE + 클라우드 코드 + 첫 배포 |
  | SMB | 3~5장 | GitOps, CI, 관측 가능성, 무중단 배포 |
  | 전환기 | 6장 | 캐시, 보안, Canary |
  | 엔터프라이즈 | 7~8장 | 규모 확장, 고도화 |
  | 종합 | 9장 | GitAIOps: 살아있는 운영 표준 |
- **1.4.2 실습 환경** — 클라우드: **GKE**(무료 크레딧 $300 활용), AI: **클로드 코드**(모든 명령을 자연어로 지시), 형상 관리: **깃허브**(매 장마다 커밋·누적).
- **1.4.3 이 책의 저장소** — 두 개: `_Book_GitAIOps`(가이드·의사결정·정답 예시), `notiflex-platform`(독자가 직접 실습하며 채워가는 저장소).

#### 1.5 Notiflex 스타트업 시나리오 소개
- **1.5.1 Notiflex란?** — 실습을 관통하는 가상의 **B2B 알림 SaaS 플랫폼**. 이메일/SMS/푸시 알림을 대신 발송해주는 서비스. Go로 작성된 API 서버 중심.
- **1.5.2 성장 타임라인** — 창업(2장) → 첫 고객(3장) → 서비스 장애(4장) → 성장 시작(5장) → 전환기(6장) → 대형 계약(7장) → 대규모 운영(8장) → 돌아보기(9장). 각 성장 단계의 문제를 인프라로 해결한다.
- **1.5.3 여러분의 역할** — 독자는 Notiflex의 DevOps 엔지니어. 클로드 코드를 도구로 삼아 서비스 성장에 맞춰 인프라를 키운다.

#### 1.6 가드레일: 클로드 코드가 정확하게 동작하는 이유
- **1.6.1 CLAUDE.md와 가드레일** — 클로드 코드는 저장소 루트의 `CLAUDE.md`를 자동으로 읽어 동작한다. 가드레일 3종: ①**의사결정 가이드**(decision-guides) ②**실행 가드레일**(prompt-guardrails) ③**결과 템플릿**(result-templates).
- **1.6.2 3단계 흐름: 탐색 → 비교 → 실행** — ①탐색("모니터링 도구는 뭐가 있지?") ②비교(대안들 세션으로 비교) ③실행("ArgoCD 설치해줘"처럼 지정된 방식대로 수행).
- **1.6.3 왜 가드레일이 필요한가** — AI가 강력해도 **정확성**이 중요하다. 가드레일은 AI가 정해진 범위 안에서, 검증된 경로를 따라 동작하게 만드는 시스템. 잘못된 값 하나가 클러스터 전체에 영향을 줄 수 있기 때문.

> [!note] 한 줄 요약
> 개발자가 인프라를 직접 다뤄야 하는 시대가 왔고, 쿠버네티스가 그 공통 언어다. GitOps로 인프라를 코드로 선언·관리하되, 사람이 하던 YAML 작성·트러블슈팅·문서화의 빈자리를 AI(클로드 코드)가 채우는 것이 **GitAIOps**다. 이 책은 Notiflex라는 가상 스타트업을 키우며 이를 실습한다.
