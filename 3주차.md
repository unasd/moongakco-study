### Chapter 5. 무중단 배포

클러스터 내부에서만 접근되던 API를 외부로 노출하고, 배포 중에도 끊기지 않는 Blue/Green을 도입한다. 실행은 클로드 코드가, 지시·판단·개입은 내가 맡았다.

#### 5.2 트래픽 관리: Gateway API
- 외부 노출을 지시. 클로드 코드가 Gateway API의 gke-l7-regional-external-managed 클래스로 구성했다. proxy-only 서브넷을 만들고 외부 IP를 할당받았으며, HealthCheckPolicy로 `/health`를 헬스체크에 연결했다.
- Gateway API는 GKE 네이티브라 별도 Ingress 컨트롤러 설치가 필요 없다. Gateway는 인프라팀, HTTPRoute는 앱팀이 관리하는 식으로 관심사가 분리된다.
- 질문: gke-l7-regional-external-managed가 ALB인가? → 맞다. GKE Gateway 컨트롤러가 이 클래스를 보고 GCP 리전 외부 L7 로드밸런서를 생성한다. Azure의 Application Gateway에 대응한다.
- 질문: allowedRoutes가 Same이면 다른 네임스페이스가 경합하나? hostname은 어디서 설정하나? → Same은 같은 네임스페이스의 HTTPRoute만 허용해 경합이 없다. hostname은 보통 Gateway 리스너에서 도메인을 열고, HTTPRoute에서 세부 host와 경로를 매칭한다.
- 질문: Azure Load Balancer + Ingress Controller를 Gateway로 바꾸면 인프라 변경과 비용은? → GKE는 Gateway 컨트롤러가 내장이라 무설치지만, GCP L7 로드밸런서가 생성돼 과금된다. 기존 라우팅을 Gateway·HTTPRoute로 재작성해야 한다. 전환 가치 판단은 느낀점 1에 적었다.

#### 5.3 무중단 배포: Argo Rollouts Blue/Green
- Blue/Green 배포를 지시. 클로드 코드가 Argo Rollouts v1.9.1을 컨트롤러와 kubectl 플러그인으로 설치하고, Deployment를 Rollout으로 전환했다. autoPromotionSeconds=30으로 30초 뒤 자동 승격되며, notiflex-api-preview 서비스로 새 버전을 미리 확인한다.
- v0.2.0을 수동 빌드해 Blue/Green 동작을 시연했다. 여기서 두 가지가 막혔다.
  - 시연용 수동 배포 커밋이 CI 트리거 경로 `.github/workflows/ci.yaml`에 걸려, CI가 SHA 태그로 재빌드·배포하면서 Blue/Green이 한 번 조용히 더 일어났다. 커밋 메시지에 `[skip ci]`를 넣어 차단했다.
  - Git Bash에서 gcloud를 호출하니 출력이 "Python"만 나오고 실제로 실행되지 않았다. Windows에선 gcloud를 PowerShell에서 실행해야 정상 동작한다.

### Chapter 6. Enterprise를 위한 기반 정비

캐시로 Pod 간 상태를 공유하고, 시크릿을 키 없이 관리하고, 배포 전략을 Canary로 진화시킨다. 실행은 클로드 코드가, 지시·판단·개입은 내가 맡았다.

#### 6.1 캐시: Valkey
- Pod가 여러 개라 `/id` 카운터가 인메모리면 Pod마다 값이 달라진다. Valkey로 카운터를 공유하도록 지시. 클로드 코드가 Bitnami Helm으로 Valkey standalone을 설치하고, 앱을 v0.3.0으로 고쳐 카운터를 Valkey INCR로 옮겼다. 연결 재시도는 10회. Pod를 재시작해도 카운터가 유지되는 것을 확인했다.
- CPU가 빠듯해 앱 replicas를 2에서 1로 줄였다.
- 질문: SSPL이 뭐고, 실무에서 쓰는 Azure Cache for Redis는 계속 써도 되나? → SSPL은 Redis가 2024년 채택한 라이선스로, 매니지드 서비스로 재판매하는 것을 제약한다. 앱에서 Redis를 쓰는 건 무관하고, Azure Cache for Redis는 Microsoft가 라이선스를 처리한 매니지드라 그대로 쓸 수 있다. Valkey는 SSPL 이전의 BSD 포크라 이 제약이 없다.

#### 6.2 시크릿 관리: Workload Identity + Secret Manager CSI
- 시크릿을 키 없이 관리하도록 지시. 클로드 코드가 클러스터·노드풀에 Workload Identity를 켜고, GKE managed Secret Manager CSI로 valkey 비밀번호를 Secret Manager에서 파일로 마운트했다. 앱은 VALKEY_PASSWORD_FILE 경로에서 읽는다. GSA notiflex-secrets에 secretAccessor와 WI 바인딩을 부여했고, SA 키 없이 인증되는 것을 확인했다.
- 이 방식이 3장에서 SA 키가 org policy에 막혔을 때 "더 정석"이라 남긴 WIF다. 키 파일 없이 OIDC로 인증하고, Secret Manager가 단일 진실 소스가 된다.
- 질문: 그러면 우리가 암호화한 건 뭐고, Pod가 읽는 값은? → 비밀번호 원본은 Secret Manager에 저장되고, 우리가 직접 암호화하진 않았다. Pod는 CSI가 마운트한 파일을 VALKEY_PASSWORD_FILE 경로에서 읽는다. 핵심은 암호화가 아니라, 키 없이 접근을 위임하는 Workload Identity다.
- 여기서 막힌 것들:
  - Secret Manager API가 꺼져 있어 secrets create가 실패 → API를 enable한 뒤 재시도
  - WI 노드풀 활성화로 노드가 재생성되며 Valkey·관측 Pod이 재스케줄됐다. GKE가 노드 구성 변경을 드레인+재생성으로 적용하는 정상 동작이다.
  - WI+CSI 활성화 후 CPU 99%, Valkey가 Pending → 관측 스택 축소로도 부족해 Loki·Fluent Bit를 임시 제거하고 Valkey 요청을 50m에서 15m으로 낮췄다.

#### 6.3 Canary 전환
- 배포 전략을 Blue/Green에서 Canary로 바꾸도록 지시. 같은 Rollout CRD에서 strategy만 canary로 바꿨다. 20→50→80→100으로, 각 단계 30초 pause. Rollout을 삭제하니 ArgoCD가 재생성했다. v0.4.0·v0.4.1을 배포해 점진 전환을 시연했다.
- Canary는 stable + canary 2개 Pod가 필요한데 CPU가 99%라, monitoring 스택을 임시로 scale=0해 자원을 확보한 뒤 시연했다.

## 느낀점 & 회고

**1. Gateway API의 사상 — 인프라팀·앱팀 관심사 분리**
Gateway API가 왜 이렇게 설계됐는지 이해할 수 있어 좋았다. Gateway는 인프라팀, HTTPRoute는 앱팀이 관리하는 관심사 분리가 설득력 있었다. 다만 L4 + Ingress Controller를 쓰는 입장에서, ALB + Gateway로 바꿀 때 따르는 인프라 변경과 비용 추가를 생각하면 아직 그만한 이득이 보이지는 않았다.

**2. 회사의 배포 방식과 비교**
회사에선 Rolling update를 쓴다. 배포 중 몇 건의 오류가 치명적으로 작용하는 시스템이 없기 때문이다. 다만 프로젝트팀이 충분히 테스트하려고 deployment + service + ingress를 한 세트 더 만들고, 테스트 완료 후 swap한 적은 있다. 말하자면 수동 Blue/Green이다. 그 수동 swap이 Blue/Green의 원형인 셈이니, 내 실무에도 필요한 부분이 있음을 깨달았고 도입을 고려할 만하다.

**3. 자동 승격 vs 수동 승격**
실무라면 수동 승격이 맞을 것 같다. 자동 30초 동안 검증할 수 있는 게 많지 않다. 결국 승격 기준은 '시간'이 아니라 '지표'여야 한다. 자동 타이머 대신 수동 승인 게이트를 두거나, 에러율·지연 같은 메트릭으로 승격 조건을 거는 방식이 안전하다.

**4. 혼자였다면 못 잡았을 트러블**
CI가 Blue/Green 시연 중 조용히 끼어든 문제는 혼자였다면 못 잡았을 것이다. 내가 직접 설정하지 않다 보니 책과 다른 CI 설정을 만든 것조차 기억하지 못했고, 기억한다 해도 코드를 들여다볼 엄두가 나지 않는다. AI가 문제를 빨리 잡아주는 만큼 내가 시스템을 이해하는 총량은 줄어든다. 편의와 이해도는 트레이드오프라, AI에 맡기더라도 무엇을 왜 바꿨는지는 사람이 남기고 검토해야 한다. JOURNEY.md와 아키텍처 스냅샷 같은 기록이 그 간극을 메우는 장치다.

**5. 가드레일은 지뢰 지도**
5.2 가드레일에는 "HealthCheckPolicy를 반드시 명시하라"가 있다. 자주 밟는 지뢰를 미리 적어둔 지도 같은 것이다. 이런 노하우를 공유하는 것이 실무 생산성을 높이는 좋은 방법이라, 내 실무 노하우도 이 방식으로 정리하고 싶다.

**6. 키를 숨기는 메커니즘**
Workload Identity + Secret Manager로 키 파일 없이 시크릿을 마운트하는 걸 보며, 키를 노출하지 않고 인증하는 메커니즘을 알 수 있어 좋았다. 3장에서 SA 키 대신 정석이라 봤던 WIF 방식이다.
